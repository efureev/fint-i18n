# Техническое задание (ТЗ) на реализацию пакета `@feugene/fint-i18n`

## 1. Цели и задачи

Разработка легковесной, высокопроизводительной библиотеки для локализации Vue 3 приложений и библиотек компонентов,
обеспечивающей гибкость в управлении ресурсами и асинхронную загрузку.

## 2. Ключевые требования

### 2.1. Производительность и размер

- **Минимальный размер**: Вес основного ядра (bundle size) должен составлять менее 2 КБ (gzip) и стремиться к абсолютному минимуму.
- **Zero Dependencies**: Максимальное отсутствие внешних зависимостей (только `vue` как peerDependency).
- **Мгновенный init**: Инициализация не должна требовать сложных вычислений или создания тяжелых виртуальных машин.
- **Сверхбыстрый `t`**:
    - Использование **прекомпиляции шаблонов** (JIT). При загрузке чанка сообщений строки-шаблоны (например,
      `"Привет, {name}!"`) преобразуются в оптимизированные функции-генераторы `(p) => "Привет, " + p.name + "!"`.
    - Кэширование скомпилированных функций для повторного использования.
    - Избегание регулярных выражений в рантайме при вызове `t`.

### 2.2. Асинхронность и блоки (Translation Blocks)

- **Асинхронные источники**: Поддержка динамических импортов для локалей.
- **Блочная структура**: Возможность разделения переводов на независимые блоки (например, `common`, `auth`, `profile`).
- **Динамическая подгрузка**: API для загрузки блоков "на лету" без перезагрузки страницы.
- **Параллельная загрузка**: Поддержка одновременной загрузки нескольких блоков/локалей через `Promise.all`.

### 2.3. DX (Developer Experience) и TypeScript

- **Простое API**: Минималистичный интерфейс, вдохновленный лучшими практиками `i18n`.
- **Строгая типизация**: Автоматическая генерация типов для ключей на основе JSON-структур (через Template Literal
  Types).
- **Vue Integration**: Поддержка через `app.use()` и композабл `useFintI18n()`. Глобальное свойство `$t` (опционально).

### 2.4. Совместимость с SPA/SSR

- **Универсальность**: Полная поддержка Single Page Applications и Server-Side Rendering (Node.js, Edge Runtime).
- **Изоляция**: Отсутствие жестких привязок к глобальным объектам браузера (`window`, `document`, `localStorage`) в ядре. Весь платформо-зависимый функционал (например, `PersistencePlugin`) выносится в плагины или адаптеры.

### 2.5. BridgeMode (Совместимость с vue-i18n)

- **Проксирование**: Если в основном приложении уже используется `vue-i18n`, `fint-i18n` должен уметь:
    - Использовать `locale` из `vue-i18n`.
    - Перенаправлять вызовы `t` в `vue-i18n.t`.
    - Синхронизировать состояние (смена языка в одном месте меняет его везде).

Этот функционал реализуется через **систему плагинов** (см. раздел 6). Если нет необходимости в совместимости, плагин не подключается, сохраняя минимальный размер ядра.

### 2.6. JIT vs AOT Compilation (Архитектурное решение)

В процессе проектирования был проведен сравнительный анализ подходов к компиляции шаблонов:

| Характеристика | JIT (Runtime, текущий) | AOT (Build-time) |
| :--- | :--- | :--- |
| **Производительность (Init)** | Минимальная задержка при загрузке чанка | Мгновенно |
| **Производительность (t)** | Очень высокая (O(N) по частям строки) | Экстремальная (нативный JS) |
| **Размер бандла ядра** | + ~300 байт (компилятор) | - 300 байт |
| **Размер файлов переводов** | Минимальный (JSON) | Выше (JS функции) |
| **Гибкость** | Любые источники (API, БД) | Только статические файлы |
| **Сложность внедрения** | Нулевая | Требует Vite-плагин |

**Решение**: Использование **JIT-компиляции** с обязательной **прекомпиляцией при загрузке блока**. Это обеспечивает оптимальный баланс: библиотека остается независимой от инструментов сборки и поддерживает динамические источники данных, при этом не теряя в скорости рендеринга.

## 3. Архитектура и API

### 3.1. Создание инстанса

```typescript
const i18n = createFintI18n({
  locale: 'en',
  fallbackLocale: 'en',
  loaders: [appLocaleLoaders, dsLocaleLoaders]
})
```

Где каждая package collection имеет вид:

```typescript
type LocaleBlockLoader = () => Promise<any>
type LocaleBlockLoaders = LocaleBlockLoader | LocaleBlockLoader[]
type LocaleLoaderCollection = Record<Locale, Record<string, LocaleBlockLoaders>>
```

Правила:

- package collections merge-ятся слева направо;
- одинаковые block keys складываются в последовательность loaders;
- сначала ищется точный block, затем ближайший parent block;
- сообщения последующих loaders переопределяют совпадающие ключи предыдущих loaders.

### 3.2. Работа с блоками

```typescript
// Загрузка блока
await i18n.loadBlock('auth');

// Проверка загрузки
if (i18n.isBlockLoaded('auth')) {
  // ...
}
```

### 3.3. Использование в компонентах

```typescript
const { t, locale } = useFintI18n();
const label = t('auth.login.submit', { name: 'User' });
```

## 4. План реализации

### Этап 1: Core Engine
- Реактивное хранилище сообщений.
- JIT-компилятор шаблонов.
- Система плагинов (ядро и хуки).

### Этап 2: Loader & Blocks
- Система регистрации загрузчиков.
- Механизм блоков и их слияния.
- Composable `useI18nScope` для управления жизненным циклом блоков.

### Этап 3: Vue Integration
- Плагин для Vue 3.
- Реализация директивы `v-t`.
- Базовые плагины: `PersistencePlugin`, `LoggerPlugin`.

### Этап 4: Advanced Features & Tooling
- `BridgePlugin` для совместимости с `vue-i18n`.
- Типизация ключей.
- Документация и примеры.

## 5. Расширенные возможности (implementation details)

### 5.1. Persistent Storage (постоянное хранилище)

Реализуется как встроенный механизм или плагин для автоматического сохранения и восстановления выбранной локали.

- **Storage Adapter**: Абстракция над хранилищем (по умолчанию `localStorage`). Интерфейс: `{ get(key): string, set(key, val): void }`.
- **Инициализация**: При вызове `createFintI18n` приоритет выбора локали: `options.locale` -> `Storage` -> `fallbackLocale`.
- **Синхронизация**: Реактивный `watch` за `locale`, который вызывает `storage.set()`.
- **Cross-Tab Sync**: Использование события `window.onstorage` для мгновенного обновления локали во всех вкладках без перезагрузки.

### 5.2. Reactive Scope Context (локальные контексты)

Позволяет компонентам объявлять свои зависимости от конкретных блоков переводов.

- **Composable `useI18nScope(blocks: string[])`**:
    - При вызове увеличивает счетчик использования указанных блоков.
    - Автоматически вызывает `loadBlock` для каждого блока при инициализации.
    - Возвращает локальный `t`, который может иметь предустановленный префикс.
    - При уничтожении компонента (`onUnmounted`) уменьшает счетчик. Если счетчик блока становится 0, сообщения могут быть выгружены из памяти (опционально для экономии ресурсов в больших приложениях).

### 5.3. Directives support (директивы Vue)

Директива `v-t` для декларативного использования в шаблонах.

- **Синтаксис**: `<span v-t="'auth.title'"></span>` или `<span v-t="{ path: 'auth.greet', params: { name: 'User' } }"></span>`.
- **Производительность**: Прямое обновление `textContent` элемента минуя часть виртуального дома Vue для простых строк.
- **Модификаторы**:
    - `.once`: Рендерит один раз и больше не следит за изменениями (максимальный перформанс для статики).
    - `.preserve`: Сохраняет текущее содержимое, если ключ не найден.

## 6. Система плагинов

Для обеспечения модульности и сохранения минимального размера ядра, `fint-i18n` использует событийно-ориентированную систему плагинов.

### 6.1. Интерфейс плагина
```typescript
interface FintI18nPlugin {
  name: string;
  install: (instance: FintI18nInstance) => void;
}
```

### 6.2. Точки расширения (Hooks)
Плагины могут подписываться на внутренние события инстанса:
- `beforeInit`: Настройка начальных параметров.
- `onLocaleChange(newLocale)`: Реакция на смену языка.
- `beforeLoadBlock(blockName)`: Пре-процессинг перед загрузкой.
- `afterLoadBlock(blockName, messages)`: Пост-процессинг (например, дополнительная трансформация сообщений).
- `onMissingKey(key, locale)`: Обработка отсутствующих ключей (логирование, внешние API).
- `onTranslate(key, params, result)`: Перехват результата перевода (для пост-фильтров или отладки).

### 6.3. Функционал, вынесенный в плагины
1. **BridgePlugin**: Реализует `BridgeMode` через проксирование `t` и синхронизацию `locale` с `vue-i18n`.
2. **PersistencePlugin**: Инкапсулирует работу с `localStorage`.
3. **LoggerPlugin**: Обработка `onMissingKey` в dev-режиме.

### 6.4. Независимость и модульность

- **Standalone Core**: Основное ядро должно полноценно функционировать без системы плагинов.
- **Build-time Optimization**: Система плагинов должна быть спроектирована так, чтобы при их неиспользовании код системы хуков и регистрации плагинов мог быть полностью исключен из сборки (Tree Shaking).
- **Минимальный размер плагинов**: Каждый плагин должен быть самодостаточным и максимально легковесным (менее 0.5 КБ gzip), обеспечивая "pay-only-for-what-you-use" модель.
